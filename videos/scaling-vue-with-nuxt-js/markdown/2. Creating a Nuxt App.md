## Creating our Initial Nuxt.js App

To get started creating our application we‚Äôll be using `create-nuxt-app`. Create Nuxt App is a command line tool that helps you create and scaffold out your Nuxt applications. It helps you set up the default folder structure of your app, you can optionally install your preferred server-side framework like Express or Koa, and you can also install the Nuxt Axios module for easy Axios integration with your app.

To get started, make sure you‚Äôre using a version of npm that‚Äôs 5.2.0 or higher. This will ensure you have have `npx` installed, we‚Äôll need this command.

Then let‚Äôs run the following in your command in the terminal:

```
    $ npx create-nuxt-app real-world-nuxt

```

**npx** is a tool intended to help round out the experience of using packages from the npm registry ‚Äî the same way npm makes it easy to install and manage dependencies hosted on the registry, npx makes it easy to use CLI tools and other executables hosted on the registry. In our case, that‚Äôs `create-nuxt-app`.

If Yarn is your package manager of choice you can run `yarn create nuxt-app real-world-nuxt`. It‚Äôll set you up in the same way.

After running either the `npx` or `yarn` command, Create Nuxt App will prompt you with a few questions to get you setup with your app‚Äôs default configuration.

Here is how we‚Äôre going to answer for our project, and I really encourage you to code with me.

It will also create a git repository for you, and run `npm install` to fetch all dependencies.

To run the app that was just created in development mode, we‚Äôll need to `cd` into the directory it created for us run the following command:

This will startup our Nuxt.js server in development mode. By default, the project will serve from `http://localhost:3000/`. Visiting that URL will show you the scaffolded project with a link to Nuxt‚Äôs documentation.

## The Nuxt.js Component Folder Structure

Open the project generated by `create-nuxt-app` in your code editor of choice, and you‚Äôll see a list of folders where your entire Nuxt application will live. You‚Äôll first notice that Nuxt.js has no `/src` directory, these folder are in root. Three of the eight folders there are the component folders:

![](https://firebasestorage.googleapis.com/v0/b/vue-mastery.appspot.com/o/flamelink%2Fmedia%2F1578373061939_3.jpg?alt=media&token=2475f5b7-591f-46b9-9087-1aea19903af5)

Each of these folders will contain component `.vue` files, and they each start out with a single default generated file, which together showed us the page we saw when we launched the development server.

![](https://firebasestorage.googleapis.com/v0/b/vue-mastery.appspot.com/o/flamelink%2Fmedia%2F1578373064141_4.jpg?alt=media&token=40f2e214-094f-4140-91aa-e35ac64a508f)

Here are the remaining five folders Nuxt generated for us.

## `/store`

A folder to contain all of your app‚Äôs Vuex Store Files. Nuxt gives you two different ways to create your store and we‚Äôll dive deeper into how as the course progresses.

## `/static`

Use this directory for storing static assets. For example, `robots.txt` or your favicon. Every single file in this directory is mapped to the server root, usually at `/`.

## `/assets`

This directory contains un-compiled assets such as Stylus or Sass files, images, or fonts. By default, Nuxt uses vue-loader, file-loader and url-loader webpack loaders for strong assets serving. If you don‚Äôt want assets to be affected by webpack, use the `static` directory for storing those assets.

## `/plugins`

This folder contains your JavaScript plugins that you want to run before instantiating the root Vue.js Application. This directory is helpful when using your own libraries or Vue plugins.

## `/middleware`

Middleware lets you define custom functions that can be run before rendering either a page or a group of pages (layouts). This folder contains your application middleware for that purpose.

Finally, you‚Äôll notice a configuration file called `nuxt.config.js`. We‚Äôll get to this in a future lesson. But, for now, know that this is the single file that you‚Äôll be using to write extra configuration, or to modify configuration that Nuxt sets up for your app by default.

## Image Assets Example

Usually we‚Äôll want to place all our images in the `assets` directory where Nuxt will use vue-loader, file-loader, and url-loader for effective asset serving. For example, if we placed a `logo.png` file in our assets directory, to use this in a component template we would write:

üìú **/pages/index.vue**

```html
    <img src="~/assets/logo.png">

```

**When we build our project, if our image is >= 1 kb**, it will use version hashes for caching and render out:

```html
    <img src="/_nuxt/img/82f7965.png">

```

The `82f7965` is the hash we‚Äôre talking about. This is beneficial because if our `logo.png` file changes in the future, but the name itself remains the same, the hash will change, and thus the new logo will be loaded in our customer‚Äôs browsers. Without the change of hash our customer‚Äôs browser may continue to load the old image.

**When we build our project, if our image is < 1 kb** it will inline the image to reduce http requests, looking something like this:

```html
    <img src="data:image/png;base64,R0l...">

```

Obviously where I put `‚Ä¶` it would be a much longer string, which contains all the data for the image. By placing the image code right there on the screen, we avoid an extra network request to fetch the file.

```javascript
        }
      ]
    })

```
